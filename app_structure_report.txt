ShiftMgmt App Structure Report
Generated from source inspection on 2026-03-09

Overview

- The active application is the Expo/React Native app under the "app" folder.
- The "server" folder exists, but "server/src" is empty, so there is no active backend implementation in this repo right now.
- The app is local-first. It uses SQLite for employees, clock events, and settings, and stores captured photos in the device document directory.
- App startup initializes the database, ensures the default admin PIN exists, ensures the photo directory exists, and then mounts the app navigator.

Top-Level Structure

- app
  - Expo/React Native frontend
  - Screens, navigation, local database, payroll logic, export logic
- server
  - Present but effectively unused in current code

Core App Architecture

- Entry point: app/App.tsx
- Navigation: app/src/navigation/RootNavigator.tsx
- Database: app/src/services/db/database.ts
- Database migrations: app/src/services/db/migrations.ts
- Employee repository: app/src/services/repositories/employeeRepository.ts

- Settings repository: app/src/services/repositories/settingsRepository.ts

- CSV export: app/src/services/export/csvExport.ts
- PDF payroll export: app/src/services/export/payrollReportExport.ts

Database Structure

SQLite tables are created and evolved in migrations:

- employees
  - basic identity
  - active/inactive flag
  - hashed PIN
  - plaintext PIN code column also exists
  - payroll/contact fields such as job title, hourly rate, department, start date, email, address, phone, and optional photo path
- clock_events
  - employee id
  - IN or OUT type
  - timestamp
  - photo path
  - source (MANUAL or AUTO)
  - admin tag (NONE, CREATED, EDITED)
  - last edited timestamp
- settings
  - key/value table used for admin PIN, payroll config, property info, auto clock-out settings, and overtime settings

Routes in the main stack:

Important navigation behavior:

- Initial route is Home.
- Admin routes are only registered after admin authentication succeeds.
- Admin logout resets navigation back to Home.

Session and Timing Rules

- Default admin PIN: 0000
- Admin PIN length: 4 digits
- Employee PIN length: 4 digits
- Admin session timeout: 2 minutes of inactivity
- Confirmation screen auto-return: 5 seconds
- Auto clock-out sweep interval on Home: every 60 seconds

Screen-by-Screen Report

1. Home Screen
File: app/src/screens/HomeScreen.tsx

Purpose:
- Main kiosk screen for staff to clock in and clock out.

Visible UI:
- Page header with app name
- Camera status badge
- Current date/time row
- Property name display
- Hidden admin hold-progress bar
- PIN entry area with PIN dots
- Error message area
- Numeric keypad
- Inline live front-camera preview
- Two action buttons:
  - Clock In
  - Clock Out

Functionality:
- Loads property name from settings.
- Requests camera permission if not already available.
- Shows live camera preview when focused.
- Runs auto clock-out sweep on focus and then every minute.
- Accepts a 4-digit employee PIN through the keypad.
- Validates the employee PIN against active employees only.
- Enforces shift transition rules:
  - employee cannot clock in twice in a row
  - employee cannot clock out unless currently clocked in
- On valid action, navigates to Camera Capture with employee id, name, and event type.

Buttons and interactions:
- Numeric keypad digits:
  - append digits up to 4 digits
- CLR:
  - clears the PIN
- DEL:
  - removes the last digit
- Clock In:
  - enabled only when 4 digits are entered and not already processing
  - validates PIN and shift state
  - routes to camera capture
- Clock Out:
  - same validation flow as Clock In
- Time row:
  - hidden admin entry point
  - user must hold for 3 seconds
  - while holding, progress bar fills and countdown text is shown
  - on completion, app vibrates briefly, clears PIN/error, and opens Admin Access

Popups/modals:
- No custom modal
- No native alert on this screen

Calendar/time controls:
- Displays current time
- No editable calendar picker here

Auto Clock-Out Behavior

File: app/src/services/clock/autoClockOutService.ts

- Controlled by settings:
  - autoClockOutEnabled
  - autoClockOutHours
- Home screen runs the sweep on focus and every 60 seconds.
- If an employee is still clocked in beyond threshold, the app creates an OUT event automatically.
- These events use a marker photo path instead of a real photo.
- Logs screen shows these rows with an AUTO marker thumbnail.

Security Notes

- PIN verification uses PBKDF2-SHA256 with 10,000 iterations.
- Admin PIN is stored hashed in settings.
- Employee PINs are hashed for verification.
- However, the employees table also stores a plaintext "pin_code" value, and the employee list can reveal it when Show PINs is enabled.

Important Observations and Gaps

1. Property settings mismatch
- The Settings screen stores propertyName, propertyAddress, and propertyDetails.
- The Home screen currently reads and displays only propertyName.
- Property address and property details appear unused in the visible UI.

2. Danger Zone is not truly "all settings"
- The delete flow removes employees and clock events, deletes photos, and resets the admin PIN.
- It does not clear payroll settings or property settings from the settings table.

3. No backend
- All operational behavior is local on the device.
- There is no active server-side syncing, auth service, or reporting API in this repo.

4. No image detail popup in logs
- The logs screen shows thumbnails/markers only.
- There is no full-screen image viewer or event detail modal.

5. No automated test suite in app source
- I did not find project tests outside dependency folders.
